# DeskGo v0.1.3 release notes

## Release intent

`v0.1.3` turns the recent homepage cleanup, Linux input hardening, Wayland bootstrap work, and demo-domain switch into a release that end users can actually consume without depending on unreleased `master` fixes.

This patch release focuses on four themes:

- making the homepage read like a product site for end users instead of a technical dashboard
- hardening Linux mouse/keyboard control without regressing Windows or macOS behavior
- improving Wayland support by auto-bootstrapping `ydotool` / `ydotoold` when possible
- switching all shipped demo/default URLs to `deskgo.ystone.us`

## Highlights

### 1. Homepage is now end-user oriented

- rewrote the Chinese and English landing pages around what users actually want to do:
  - open a desktop from the browser
  - download the app
  - set up autostart
- added quick-copy buttons for the shell and PowerShell install snippets
- moved self-hosting and advanced relay details out of the primary path so the homepage is less technical

### 2. Linux input is much more reliable

- browser input events now send explicit keyboard up/down state plus browser `code`
- mouse-button bitmask handling now matches browser left/right/middle semantics correctly
- wheel events now use tap-style button 4/5 behavior instead of sticky state emulation
- Linux pointer movement now goes through XTEST motion injection
- display-origin-aware coordinate mapping prevents pointer drift on offset-monitor layouts
- high-frequency stream logs were reduced to periodic summaries so input diagnostics stay readable

### 3. Wayland sessions now bootstrap better

- when Wayland is detected and `ydotool` / `ydotoold` are missing, the Linux client now attempts package-manager installation through:
  - `apt-get` / `apt`
  - `dnf`
  - `yum`
  - `pacman`
  - `zypper`
  - `apk`
- automatic bootstrap supports direct root execution, passwordless `sudo` / `doas`, and desktop-session `pkexec`
- the client now auto-detects or starts `ydotoold`, chooses a reusable socket path, and retries failed ydotool commands after daemon repair
- Windows and macOS input code paths remain isolated from these Linux-specific changes

### 4. Demo and default URLs now use the new hostname

- the default Desktop CLI relay URL now points to `wss://deskgo.ystone.us/api/desktop`
- README examples, autostart script defaults, and proxy tests were updated to the new hostname
- the public demo URL shown in docs now points to `https://deskgo.ystone.us`

## Validation

Validated before release with:

- `go test ./cmd/client ./cmd/relay ./internal/relay`
- `go test -tags desktop ./cmd/client`
- `go build ./cmd/relay`
- `go build -tags desktop -o /tmp/deskgo-client-macos ./cmd/client`
- `GOOS=windows GOARCH=amd64 CGO_ENABLED=0 go build -tags desktop -o /tmp/deskgo-client-windows.exe ./cmd/client`
- `GOOS=linux GOARCH=amd64 CGO_ENABLED=0 go build -tags desktop -o /tmp/deskgo-client-linux ./cmd/client`
- `GOOS=linux GOARCH=arm64 CGO_ENABLED=0 go build -tags desktop -o /tmp/deskgo-client-linux-arm64 ./cmd/client`
- `docker run --rm -v "$PWD":/src -w /src golang:1.24 go test -tags desktop ./cmd/client`
- `bash -n scripts/deskgo-autostart.sh`
- PowerShell parser validation for `scripts/deskgo-autostart.ps1`
- repository-wide verification that no `deskgo.zty8.cn` references remain

## Known limitations

- Windows desktop capture/input still needs more real-device runtime coverage outside this environment
- Wayland-native control still depends on the host allowing `ydotool` / `ydotoold` access to the required input devices
- relay deployments are still best placed behind a reverse proxy, tunnel, or another access-control layer
